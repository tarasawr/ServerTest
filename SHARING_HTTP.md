# Sharing HTTP API (`projects.js`)

Описание REST-эндпоинтов реестра шаринга проектов. Реализация: [`projects.js`](./projects.js), подключение из [`server.js`](./server.js) через `handleRequest`. Утилитарные/отладочные эндпоинты живут прямо в `server.js` (см. раздел в конце).

## Общее

- **Базовый URL**: тот же хост и порт, что у HTTP-сервера. По умолчанию `http://localhost:3000` (порт берётся из `process.env.PORT`, иначе `3000`). Прод-инстанс: `https://servertest-1qjf.onrender.com`.
- Заголовок для тел JSON: `Content-Type: application/json`.
- Ошибки: JSON `{ "error": "message" }` с соответствующим HTTP-статусом.
- **Хранилище — PostgreSQL (Neon)**, не in-memory: данные переживают перезапуск сервера. Схема и подключение — в [`db.js`](./db.js). Строка подключения обязательна в переменной окружения `DATABASE_URL` (без неё процесс падает на старте). Таблицы (`projects`, `project_users`) создаются автоматически при первом запуске.

## Модель ролей

Каждый проект имеет **глобальную роль** (`globalRole`) — роль по умолчанию для участников без индивидуальной.
У участника индивидуальная роль может быть задана или отсутствовать (`null` в БД → наследует `globalRole`).
**Эффективная роль** = `owner` (для владельца) → индивидуальная роль (если задана) → `globalRole` проекта.

Все listing-эндпоинты возвращают уже **вычисленную эффективную роль** — клиент никогда не получает `null`.

| Значение роли | Описание |
|---------------|----------|
| `owner` | Владелец (создатель проекта). Роль владельца изменить/удалить нельзя |
| `can_edit` | Может вносить изменения |
| `can_view` | Только просмотр (значение `globalRole` по умолчанию) |
| `no_access` | «Мягко удалённый» участник: строка сохраняется, но пользователь скрыт из всех списков, а повторный заход по ссылке отклоняется (`POST /join` → `403 no_access`) |

Дополнительно у участника есть:
- `isInvitationPending` — приглашён владельцем, но ещё не заходил по ссылке (см. `POST .../invite`).

Цвет аватара сервер не назначает и не отдаёт — он вычисляется на клиенте детерминированно из `userId`.

## Авторизация write-операций

Все изменяющие эндпоинты «только для владельца» требуют в теле поле **`callerUserId`**, которое сервер сверяет с `projects.owner_user_id` (`assertIsOwner`). При несовпадении/отсутствии — `403`, при отсутствии проекта — `404`.

## Эндпоинты

### `POST /projects`

Регистрация проекта в реестре. Идемпотентна: повтор с тем же `projectId` обновляет `projectXml`, `projectTitle` (если непустой) и `lastSyncDate`, владельца НЕ меняет. Реализована атомарным UPSERT.

**Тело:**

| Поле | Обязательно | Описание |
|------|-------------|----------|
| `projectId` | да | Строковый идентификатор проекта |
| `ownerUserId` | да | ID владельца |
| `ownerName` | нет | Имя владельца (по умолчанию `Unknown`) |
| `ownerAvatarUrl` | нет | URL аватара владельца (обновляется при ре-регистрации владельцем) |
| `projectXml` | нет | XML-снапшот проекта |
| `projectTitle` | нет | Заголовок проекта |

**Успех (200):** `{ "ok": true, "projectId": "...", "shareUrl": "/projects/{projectId}" }`
`shareUrl` — **относительный** путь; полная ссылка = база + `shareUrl`. Новый проект создаётся с `globalRole: "can_view"`, а владелец сразу добавляется в `project_users` с ролью `owner`.

**Ошибки:** `400` — нет `projectId` или `ownerUserId`.

---

### `GET /projects/user/:uid`

Список проектов, в которых участвует пользователь `:uid` (сегмент URL-декодируется). Скрываются проекты, где у пользователя роль `no_access`. Сортировка — по `lastSyncDate` (свежие сверху).

**Успех (200):** `{ "projects": [ { "projectId", "ownerName", "projectTitle", "role", "lastSyncDate" }, ... ] }`
`role` — эффективная роль пользователя в каждом проекте.

---

### `POST /projects/:id/join`

Добавить пользователя в проект. Новый участник добавляется **без индивидуальной роли** (наследует `globalRole`). Уже существующий — получает свою текущую эффективную роль; при наличии флага приглашения он снимается (`isInvitationPending → false`).

**Тело:**

| Поле | Обязательно | Описание |
|------|-------------|----------|
| `userId` | да | ID пользователя |
| `name` | нет | Отображаемое имя |
| `avatarUrl` | нет | URL аватара |

**Успех (200):**
- новый участник: `{ "ok": true, "role": "<globalRole проекта>" }`
- уже в проекте: `{ "ok": true, "role": "<эффективная роль>", "alreadyMember": true }`

**Ошибки:** `404` — проект не найден; `400` — нет `userId`; `403 { "error": "no_access" }` — доступ отозван владельцем (ссылка для этого пользователя больше не действует).

---

### `POST /projects/:id/leave`

Самостоятельный выход участника из проекта (жёсткое удаление строки). Владелец выйти не может.

**Тело:** `{ "userId": "<id уходящего>" }`

**Успех (200):** `{ "ok": true }`

**Ошибки:** `404` — проект не найден или пользователь не в проекте; `400` — нет `userId` или попытка выхода владельца.

---

### `GET /projects/:id`

Метаданные проекта (без `projectXml` и **без** `ownerUserId` — клиент держит ownership локально; сервер остаётся источником истины для write-проверок).

**Успех (200):** `{ "projectId", "ownerName", "lastSyncDate", "globalRole", "userCount" }`
`userCount` не учитывает участников `no_access`.

**Ошибки:** `404` — проект не найден.

> Используется и как лёгкая проба доступности сервера (C# `IsServerReachable`).

---

### `GET /projects/:id/users`

Список всех участников с доступом (скрыты `no_access`).

**Успех (200):**
```json
{
  "globalRole": "can_view",
  "users": [
    { "userId", "name", "avatarUrl", "role", "isInvitationPending" },
    ...
  ]
}
```
`role` — эффективная роль каждого участника (никогда не `null`).

**Ошибки:** `404` — проект не найден.

---

### `GET /projects/:id/active-users`

Пользователи, **прямо сейчас** находящиеся в активной WebSocket-сессии, привязанной к проекту (в отличие от `/users` — все, у кого есть доступ). Если привязанной сессии нет или в ней нет игроков — пустой список.

**Успех (200):**
```json
{ "users": [ { "userId", "name", "avatarUrl" }, ... ] }
```

**Ошибки:** `404` — проект не найден.

---

### `PUT /projects/:id/globalRole`

Изменить глобальную роль проекта. **Только владелец.**

**Тело:** `{ "globalRole": "can_view" | "can_edit", "callerUserId": "<id владельца>" }`

**Успех (200):** `{ "ok": true, "globalRole": "..." }`

**Побочный эффект:** всем гостям в привязанных WS-сессиях, чья роль наследует глобальную (`role IS NULL`), рассылается `{ "type": "RoleChanged", "projectId", "userId": "", "newRole": "..." }`, и их серверная роль синхронизируется (для гейта `canEdit`). Индивидуальные роли не перезаписываются.

**Ошибки:** `400` — неверная роль; `403` — не владелец; `404` — проект не найден.

---

### `PUT /projects/:id/users/:uid/role`

Установить **индивидуальную** роль участника. **Только владелец.** Нельзя менять роль владельца. Допустимы только `can_view` / `can_edit`.

**Тело:** `{ "role": "can_view" | "can_edit", "callerUserId": "<id владельца>" }`

**Успех (200):** `{ "ok": true, "role": "..." }`

**Побочный эффект:** этому пользователю в привязанных WS-сессиях рассылается `{ "type": "RoleChanged", "projectId", "userId", "newRole" }` с синхронизацией серверной роли.

**Ошибки:** `403` — не владелец; `404` — проект или пользователь не найден; `400` — неверная роль или попытка изменить роль владельца.

---

### `POST /projects/:id/users/:uid/invite`

Владелец приглашает пользователя `:uid`. Вставляет строку с `isInvitationPending = true` (роль `null`), захватывая `name`/`avatarUrl` (чтобы приглашённый отображался в списке до захода). **No-op**, если пользователь уже в проекте (флаг приглашения повторно не поднимается). **Только владелец.**

**Тело:** `{ "callerUserId": "<id владельца>", "name": "...", "avatarUrl": "..." }`

**Успех (200):** `{ "ok": true }`

**Ошибки:** `400` — нет `projectId`/`userId`; `403` — не владелец; `404` — проект не найден; `409` — попытка пригласить самого себя (владельца).

---

### `DELETE /projects/:id/users/:uid`

Удалить конкретного участника. **Только владелец.** Удаление **мягкое**: роль переводится в `no_access` (строка остаётся, чтобы отклонять повторный заход по ссылке), флаг приглашения сбрасывается.

**Тело:** `{ "callerUserId": "<id владельца>" }`

**Успех (200):** `{ "ok": true }`

**Побочный эффект:** если пользователь сейчас в привязанной WS-сессии — ему шлётся `{ "type": "UserRemoved", "projectId", "userId" }`, после чего его соединение закрывается (передача владения/`PlayerLeft` обрабатывает `leaveSession` в `server.js`).

**Ошибки:** `403` — не владелец; `400` — попытка удалить владельца; `404` — проект/пользователь не найден.

---

### `DELETE /projects/:id/users`

Удалить **всех** участников, кроме владельца (жёсткое удаление строк). **Только владелец.**

**Тело:** `{ "callerUserId": "<id владельца>" }`

**Успех (200):** `{ "ok": true, "removedCount": N }`

**Ошибки:** `403` — не владелец; `404` — проект не найден.

---

### `DELETE /projects/:id`

Удалить проект из реестра целиком. **Только владелец.** Строки `project_users` удаляются каскадом (`ON DELETE CASCADE`).

**Тело:** `{ "callerUserId": "<id владельца>" }`

**Успех (200):** `{ "ok": true }`

**Побочный эффект:** все активные WS-сессии этого проекта **принудительно завершаются**: участникам шлётся `{ "type": "SessionClosed", "reason": "sharing_stopped" }` (PascalCase — под `Enum.TryParse` на стороне Unity), соединения закрываются, сессии удаляются из `sessions`/`projectIndex`.

**Ошибки:** `403` — не владелец; `404` — проект не найден.

---

### `GET /projects/:id/xml`

Сырой XML-снапшот проекта. Ответ — **не JSON**, а `text/xml; charset=utf-8`.

**Успех (200):** тело = XML (пустая строка, если снапшота нет).

**Ошибки:** `404` — проект не найден.

> Клиент (`GetProjectXmlEx`) отличает 404/пустое тело от сетевой ошибки, чтобы при offline открыть локально кэшированную копию, а не считать проект удалённым.

---

### `GET /projects/:id/session`

Найти **активную** WS-сессию, привязанную к проекту (первая сессия с `projectId === id` и хотя бы одним игроком).

**Успех (200):** `{ "inviteCode": "..." }`

**Ошибки:** `404` — проект не найден или нет активной привязанной сессии.

> Используется клиентом после `POST /projects/:id/join`, чтобы получить `inviteCode` и подключиться к WS-сессии.

---

### `PUT /sessions/:inviteCode/project`

Привязать активную WS-сессию к зарегистрированному проекту.

**Тело:** `{ "projectId": "..." }`

**Успех (200):**
- `{ "ok": true }` — у сессии выставлен `projectId`, она проиндексирована в `projectIndex`.
- `{ "ok": true, "alreadyLinked": true }` — сессия уже привязана к этому же проекту.

**Ошибки:** `400` — нет `projectId` или попытка привязать legacy-сессию (`__legacy__`); `404` — проект не найден или сессии с таким `inviteCode` нет; `409` — сессия уже привязана к другому проекту **или** другой проект уже занят другой сессией.

## Утилитарные / отладочные эндпоинты (`server.js`)

Реализованы напрямую в `server.js`, мимо `projects.js`. Сессии и боты живут в памяти процесса.

### `GET /` (корень)
Health-check в `text/plain`: `Multiplayer server OK. Sessions: N, Clients: M`.

### `GET /sessions`
Список активных WS-сессий (legacy `__legacy__` исключена).
`{ "sessions": [ { "id", "inviteCode", "players", "linkPermission" }, ... ] }` — `players` — число подключённых.

### `GET /sessions/latest/project`
XML проекта из первой активной сессии с игроками.
`{ "inviteCode": "...", "projectXml": "..." }`; `404` — нет активных сессий с игроками.

### `GET /bots`
Управление ботами (нагрузочное тестирование). Без параметров — текущие настройки. Возвращает:
`{ botCount, viewMode, mobileMode, rejoin, sessionMin, sessionMax, offlineMin, offlineMax, floorMode, floorSwitch, behaviorChange }`.

Основные query-параметры (все опциональны, применяются немедленно):

| Параметр | Описание |
|----------|----------|
| `count=0..10` | Число ботов на сессию (`>0` — спаун во всех активных сессиях) |
| `mode=2d\|3d\|panorama\|random` | Режим просмотра ботов |
| `mobile=all\|random` | Доля «мобильных» ботов |
| `rejoin=0\|1` | Цикл отключения/переподключения |
| `sessionMin/sessionMax` | Длительность онлайна бота (сек) |
| `offlineMin/offlineMax` | Длительность офлайна перед переподключением (сек) |
| `floorMode=random\|first`, `floorSwitch=0..1` | Поведение смены этажа |
| `behaviorChange=0\|1` | Случайная смена режима просмотра |
| `reset=1` | Остановить всех ботов и (если `count>0`) переспаунить |

### `GET /rooms`
Разобранный список комнат из XML первой активной сессии (отладка):
`{ "rooms": [ { "index", "center": { "x", "z" }, "area", "indoor" }, ... ] }`.
При отсутствии сессии — `{ "rooms": [], "error": "No active session" }` со статусом `200`.

### `GET /test` / `POST /test`
Координация тестов между несколькими экземплярами Unity.
**GET** — текущее состояние (`{ "phase": "idle", ... }`). **POST** — перезаписать состояние произвольным JSON (`{ "ok": true }`; `400` при невалидном JSON).
